# Invest-Ed Security Audit — 2026-05

## Scope
Backend (FastAPI) + frontend (React/TS) app code + dependency CVEs. LLM surface: audit/document only (fixes deferred to AI sub-project). Excludes infra/DAST/pentest.

## Threat Model
### Assets
Child PII (email, dob, username, parent_email, progress); parent PII + parent-session tokens; JWT access/refresh tokens; CSRF tokens; one-time tokens (consent/verify/reset); simulator virtual state; audit logs; LLM provider API keys.
### Trust boundaries
unauth <-> authed child; child <-> parent; app <-> LLM providers; app <-> email provider.
### Actors
unauth attacker; malicious authed child; malicious authed parent (cross-family IDOR); compromised dependency; network MITM.
### Highest-impact (prioritise)
A01 access control / IDOR on /parent/*, /users/*, simulator, content; A02 secret/cookie/transport handling; A07 auth/session integrity; A09 PII/secret leakage in logs/errors.

## Severity Rubric
- **Critical**: unauth remote compromise OR mass child-PII exposure.
- **High**: authed privilege escalation / cross-family IDOR / single-user PII exposure / auth bypass.
- **Medium**: security misconfig with limited blast radius; info leak requiring chained conditions.
- **Low**: defence-in-depth gaps; hardening nits.

## LLM Surface (Audit-only - Deferred to AI sub-project)
| ID | Risk | Severity | Recommended mitigation | Status |
|----|------|----------|------------------------|--------|
| LLM-01 | Prompt injection via child free-text. Child `message` reaches the model as a discrete `{"role":"user"}` entry (`tutor_service.py:chat` L140-153; `chart_coach_service.py:chart_coach_chat` L121-133) — role separation is correct, system prompt is a separate arg — but there is NO delimiting/escaping of untrusted text and the system prompts give weak, overrideable NL guardrails ("Only explain concepts...", "Rule 1..."). A child can prompt-inject to make Coach Eddie ignore the lesson-only/no-advice rules and emit off-topic or age-inappropriate content. No exfiltration target exists on this surface (system prompt holds only lesson JSON / public chart stats, no secrets/other-user data). Realistic impact for this app = a kid eliciting unsafe/off-topic LLM content, not data theft. | Medium | Wrap untrusted lesson/chart/user text in explicit delimiters; harden system prompt against override; add an allowed-topic / refusal constraint and a server-side output check (see LLM-03); consider provider-side guardrail models. | Deferred → AI sub-project |
| LLM-02 | PII minimisation to external providers (Together/Groq/OpenAI/Anthropic). EXACT data sent per service: **tutor** (`tutor_service.py:chat` L134-153) — system prompt = static template + skill-level string + `json.dumps(lesson.content_json or {})`; messages = full prior conversation history + the child's typed message. **chart-coach** (`chart_coach_service.py` L117-133, `_build_system_prompt`/`_build_stats`) — derived integer `age = (date.today()-user.dob).days//365`, ticker/name/period, computed price stats, conversation history + child message. **ai_content** (`ai_content_service.py:generate_practice_quiz` L77-94) — topic + `json.dumps(lesson.content_json)` only. NO direct identifiers (email/dob/username/parent_email/user.id) are sent by any service; chart-coach sends a coarse derived age, not the dob. Residual: child-typed free-text (tutor/chart-coach messages) is sent verbatim and may contain self-disclosed PII the app cannot pre-strip. | Low | Keep direct-identifier exclusion as an explicit invariant + regression test; consider provider zero-retention/no-train terms and an optional PII scrub on outbound child free-text; document the coarse-age disclosure. | Deferred → AI sub-project |
| LLM-03 | Kid-safety of model output: the only filter is a narrow regex `_ADVICE_PATTERNS` ("you should buy/sell/...", "buy/sell/invest in <Cap>") in `tutor_service.py:safety_filter` (L57-73) and `chart_coach_service.py:_safety_filter` (L25-40); `ai_content` has none (relies on JSON-schema validation only). No provider safety settings, no moderation/classification pass, no allowed-topic enforcement before model text is shown to a child. The regex trivially misses paraphrase, non-English, self-harm, sexual, violent, or grooming-style content. For a financial-education app used by children this is a genuine child-safety gap. | High | Add a moderation pass (provider moderation endpoint or a safety classifier) on every model response before display; enable provider safety/guardrail settings; constrain to an allowed topic set; expand beyond the advice regex. Deferred by spec decision, not by low importance — prioritise early in the AI sub-project. | Deferred → AI sub-project |
| LLM-04 | Model-output → UI injection (XSS via model). Model text returned as `response`/`m.content` is rendered as escaped JSX text in `frontend/src/components/child/lesson/CoachEddiePanel.tsx:71` and `components/child/simulator/ChartCoachPanel.tsx:86` (`{m.content}`); no raw-HTML React prop / DOM-write sink on these paths. XSS-via-model is currently mitigated by the frontend's default escaping, not by backend output handling. | Low | Maintain the no-raw-HTML invariant on model-output render paths (covered by FE A03 regression scope); if markdown rendering is ever added, sanitise. Cross-ref: Findings Register FE A03 row (`CoachEddiePanel.tsx` / `ChartCoachPanel.tsx` render model output as escaped JSX). | Deferred → AI sub-project (mitigation already provided by FE A03 coverage) |
| LLM-05 | Provider API-key handling. Keys are sourced ONLY from `settings` (`llm_client.py` `_build_together_client`/`_build_groq_client` L218-231, `get_llm_client` L261-285); never logged, returned, or sent to the client. `FallbackLLMClient` logs only provider index + error class (`llm_client.py:188,213`). `LLMError(str(last_error))` wraps provider SDK exception strings (L84,138,107,158) and could surface to a 500 path, but API keys are not interpolated into those strings and the global handler returns generic `Internal Server Error` (per A04/A05 + A09 coverage). No key leak found. | Low | Keep keys config/env-only; add an assertion/test that `LLMError`/log messages never contain key material; ensure provider error strings are not echoed verbatim to clients. NOT a new A09 miss — consistent with the committed A09 "NO FINDING" row. | Cross-ref A09/Task 6 (no leak found; defence-in-depth deferred → AI sub-project) |
| LLM-06 | Prompt/quota/cost abuse & rate limiting. Per-conversation message cap exists (`tutor_service.py:chat` L99-126 via `tutor_max_messages_free/premium`; `chart_coach_service.py` L92-115) but it is PER-CONVERSATION — a child can open unlimited new conversations (new `conversation_id=None`) with no per-child/day quota. The LLM endpoints `POST /tutor/chat` (`ai.py:65`), `POST /lessons/{id}/practice` (`ai.py:35`) and `POST /market/chart-coach` (`simulator.py:337`) carry NO `@limiter.limit` decorator, and `app/core/rate_limit.py` constructs `Limiter(...)` with NO `default_limits` and `enabled=False` in development — so in any env these routes are not rate-limited. An authenticated child (or compromised child session) can drive effectively unbounded billed external LLM calls (cost/DoS abuse). | Medium | Add per-child daily/hourly LLM-call quota (DB-backed, not per-conversation) and `@limiter.limit` (or a tiered cost budget) on the three LLM endpoints; cap concurrent/new-conversation creation. Cross-ref: Findings Register A05-1 (CSRF on `/tutor/chat` already fixed in Task 6 — closes the cross-site abuse vector but not the authenticated quota/cost vector). | Deferred → AI sub-project |
